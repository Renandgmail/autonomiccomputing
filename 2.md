# Layer: Knowledge & Entity Model

---

## 1. Purpose

Define a **unified, canonical representation of all codebase entities and their relationships**.

This layer ensures:
- Consistency across all agents
- Composability of outputs
- Graph-ready data structure
- Search and analysis compatibility

---

## 2. Scope

### 2.1 In-Scope
- Entity definition (class, method, file, config, API, etc.)
- Global identity strategy
- Relationship modeling
- Metadata normalization
- Confidence and provenance tracking

### 2.2 Out-of-Scope
- Parsing logic (handled by extractor agents)
- UI rendering
- Query execution
- Runtime telemetry (future layer)

---

## 3. Core Concepts

### 3.1 Entity

An entity represents any identifiable unit in the system.

#### Supported Entity Types (initial)
- repository
- module
- file
- class
- interface
- method
- property
- database_table
- api_endpoint
- configuration_key

---

### 3.2 Global Entity Identity

Each entity MUST have a globally unique ID.

#### ID Format
```

<repo>::<type>::<normalized_path_or_name>::<hash>

````

#### Rules
- Deterministic generation
- Stable across runs (if source unchanged)
- Case-normalized
- Path-aware

---

### 3.3 Attributes

Entities carry structured metadata.

Examples:
- file → path, size, language
- class → name, namespace, modifiers
- method → signature, return type, parameters

---

### 3.4 Relationships

All connections between entities MUST be explicit and typed.

#### Supported Relationship Types (initial)

- contains
- defines
- depends_on
- calls
- implements
- extends
- references
- exposes
- maps_to

---

### 3.5 Relationship Rules

- MUST be directional
- MUST reference valid entity IDs
- MUST include type
- SHOULD include confidence

---

### 3.6 Confidence Model

Each entity and relationship MUST include confidence.

#### Range:
- 0.0 → low certainty
- 1.0 → absolute certainty

#### Source Priority:
1. AST (highest confidence)
2. Configuration
3. Heuristic
4. Inferred (lowest)

---

### 3.7 Provenance Tracking

Each field must indicate origin:

- AST
- Config
- Heuristic
- User-defined (future)

---

## 4. Canonical Data Model

### 4.1 Entity Schema

```json
{
  "id": "string",
  "type": "string",
  "name": "string",
  "namespace": "string",
  "attributes": {},
  "relationships": [],
  "confidence": 0.0,
  "provenance": {
    "source": "AST | CONFIG | HEURISTIC | INFERRED",
    "agent_id": "string"
  },
  "version": "string",
  "timestamp": "ISO-8601"
}
````

---

### 4.2 Relationship Schema

```json
{
  "type": "string",
  "source": "entity_id",
  "target": "entity_id",
  "confidence": 0.0,
  "provenance": {
    "source": "AST | HEURISTIC | INFERRED"
  }
}
```

---

## 5. Normalization Rules

* All names must be lowercase (except language constraints)
* Paths must be absolute within repo context
* Remove duplicates via ID matching
* Merge entities if IDs match

---

## 6. Side Effects

### 6.1 Database

* Entities stored in:

  * entities table
* Relationships stored in:

  * relationships table

#### Operation Type:

* Upsert only

---

### 6.2 Indexing

* Search index updated with:

  * name
  * type
  * attributes

* Graph index:

  * node creation/update
  * edge creation/update

---

### 6.3 Events

Emit:

* entity.created
* entity.updated
* relationship.created

---

## 7. Workflow Integration

### 7.1 Upstream

---

### 7.2 Downstream

* Search Layer
* Graph Layer (future)
* Analysis Agents
* UI Layer

## 9. Failure Handling

### Failure Conditions

* Missing entity ID
* Invalid relationship reference
* Schema violation

### Strategy

* Reject invalid entities
* Log with trace ID
* Allow partial success

---

## 10. Evaluation Criteria

### Functional

* Entity uniqueness: 100%
* Relationship validity: ≥ 99%

### Performance

* Must support large repositories (>1M LOC)

---

## 11. Constraints & Guardrails

* No duplicate entities allowed
* No implicit relationships
* All outputs must be deterministic
* Confidence scoring mandatory

---

## 12. Observability

* Track:

  * entity count
  * relationship count
  * processing time

* Logs:

  * normalization steps
  * merge operations

---

## 13. Security

* No sensitive data in attributes
* Sanitize configuration values

---

## 14. Versioning

* Schema version must be tracked
* Backward compatibility required

---

## 15. Future Enhancements

* Code Property Graph integration
* Cross-repository entity linking
* Semantic enrichment via embeddings
