# Layer: Extraction & Parsing

---

## 1. Purpose

Transform raw repository data into **structured, machine-readable representations** (AST, configs, metadata) that feed the Knowledge & Entity Model layer.

This layer is responsible for:
- Deterministic parsing
- Language-aware extraction
- Structural correctness
- Producing high-confidence, traceable outputs

---

## 2. Scope

### 2.1 In-Scope
- Source code parsing (AST generation)
- Configuration parsing (JSON, YAML, ENV, XML)
- File system metadata extraction
- Language detection
- Dependency file parsing (e.g., package manifests)

### 2.2 Out-of-Scope
- Relationship inference (handled in knowledge layer)
- Business logic interpretation
- Search indexing
- Graph construction

---

## 3. Core Components (Agent Types)

This layer is composed of specialized agents:

### 3.1 Repository Scanner Agent
- Scans file system
- Identifies file types
- Produces file inventory

---

### 3.2 Language Detection Agent
- Determines programming language per file
- Uses extension + content heuristics

---

### 3.3 AST Parser Agents (Language-Specific)

Examples:
- C# AST Parser
- Java AST Parser
- TypeScript AST Parser

Responsibilities:
- Parse source code into AST
- Extract structural nodes:
  - classes
  - methods
  - interfaces
  - imports

---

### 3.4 Configuration Parser Agents

Supported formats:
- JSON
- YAML
- XML
- ENV

Responsibilities:
- Extract key-value structures
- Preserve hierarchy
- Detect environment-specific overrides

---

### 3.5 Dependency Parser Agent

- Extracts dependencies from:
  - package.json
  - .csproj
  - pom.xml
  - requirements.txt

---

## 4. Inputs

### 4.1 Required Inputs

| Name | Type | Description |
|------|------|-------------|
| repository_path | string | Local or mounted repo path |
| commit_id | string | Snapshot reference |

---

### 4.2 Optional Inputs

| Name | Type | Description | Default |
|------|------|-------------|---------|
| file_filters | array | Include/exclude patterns | null |
| language_whitelist | array | Restrict languages | all |

---

### 4.3 Context Sources

- Git repository snapshot
- File system
- Language grammars/parsers
- Configuration schemas

---

## 5. Processing Logic (Deterministic)

### Step 1: Repository Scan
- Traverse directory
- Ignore:
  - .git
  - node_modules
  - bin/obj

---

### Step 2: File Classification
- Classify files by:
  - extension
  - MIME type

---

### Step 3: Language Detection
- Map files to languages
- Fallback to heuristic detection if unknown

---

### Step 4: AST Parsing
- Invoke language-specific parser
- Generate AST
- Extract structural nodes only (no inference)

---

### Step 5: Configuration Parsing
- Parse config files
- Preserve structure and hierarchy

---

### Step 6: Dependency Extraction
- Parse dependency manifests
- Extract:
  - package name
  - version
  - scope

---

### Step 7: Validation
- Ensure:
  - No malformed AST
  - Valid config structure

---

### Step 8: Output Preparation
- Normalize outputs
- Attach provenance
- Assign confidence

---

## 6. Output Specification

### 6.1 AST Output Schema

```json
{
  "file_path": "string",
  "language": "string",
  "entities": [
    {
      "type": "class|method|interface",
      "name": "string",
      "location": {
        "start_line": 0,
        "end_line": 0
      },
      "attributes": {},
      "confidence": 1.0,
      "source": "AST"
    }
  ],
  "imports": [],
  "confidence": 1.0
}
````

---

### 6.2 Configuration Output Schema

```json
{
  "file_path": "string",
  "config_type": "json|yaml|xml|env",
  "data": {},
  "confidence": 0.95,
  "source": "CONFIG"
}
```

---

### 6.3 Dependency Output Schema

```json
{
  "file_path": "string",
  "dependencies": [
    {
      "name": "string",
      "version": "string",
      "scope": "runtime|dev"
    }
  ],
  "confidence": 0.95
}
```

### 9.3 Edge Cases

* Unsupported language → mark file as skipped
* Corrupted file → log error, continue processing
* Large files → chunk processing

---

## 10. Failure Handling

### Failure Conditions

* Parser crash
* Invalid syntax
* Missing files

### Strategy

* Skip file, continue processing
* Log failure with file reference
* Emit failure event

---

## 11. Evaluation Criteria

### Functional

* Parsing success rate ≥ 95%
* Correct entity extraction ≥ 98%

---

### Performance

* Max processing time per file: <100ms (avg)
* Large repo support: >1M LOC

---

### Test Scenarios

* Multi-language repo
* Invalid syntax files
* Large monorepo

---

## 12. Constraints & Guardrails

* No inference beyond parsing
* No relationship creation
* Deterministic outputs only
* Language-specific parsing must be isolated

---

## 13. Observability

Track:

* Files processed
* Parsing success rate
* Error rate

Logs:

* Per-file parsing status
* Parser errors

---

## 14. Security

* Do not execute code
* Treat all files as untrusted input
* Sanitize config values

---

## 15. Versioning

* Parser version must be tracked
* Output schema versioned

---

## 16. Future Enhancements

* Incremental parsing (diff-based)
* Streaming AST generation
* Partial parsing for large files
* Integration with Code Property Graph (CFG/DFG)
