# Layer: Enrichment & Relationship

---

## 1. Purpose

Enhance parsed data with:
- Explicit relationships
- Semantic meaning
- Domain vocabulary
- Structural patterns

This layer transforms raw AST/config data into:
> **connected, queryable, semantically rich knowledge**

---

## 2. Scope

### 2.1 In-Scope
- Relationship extraction (calls, depends_on, etc.)
- Vocabulary normalization
- Pattern detection (structural + architectural)
- Semantic tagging
- Cross-entity linking

### 2.2 Out-of-Scope
- Raw parsing (Layer 2)
- Search indexing (Layer 4)
- UI rendering
- Runtime telemetry

---

## 3. Core Agent Types

---

### 3.1 Relationship Extraction Agent

#### Responsibility:
Extract explicit relationships between entities.

#### Supported Relationships:
- calls (method → method)
- depends_on (class → class/module)
- implements (class → interface)
- extends (class → base class)
- references (file → entity)
- uses_config (code → config)

---

### 3.2 Vocabulary Extraction Agent

#### Responsibility:
Standardize domain terminology.

#### Functions:
- Extract meaningful terms from:
  - class names
  - method names
  - config keys
- Normalize synonyms
- Build domain dictionary

Example:
- “UsrSvc”, “UserService” → “user_service”

---

### 3.3 Pattern Detection Agent

#### Responsibility:
Identify reusable structural and architectural patterns.

#### Pattern Types:
- Repository pattern
- Service layer pattern
- Controller pattern
- Dependency injection usage
- Layer violations

---

### 3.4 Semantic Tagging Agent

#### Responsibility:
Attach meaning to entities.

#### Tags Examples:
- business_logic
- data_access
- api_layer
- configuration
- infrastructure

---

### 3.5 Cross-Linking Agent

#### Responsibility:
Connect entities across domains:
- Code ↔ Config
- Code ↔ Database
- API ↔ Implementation

---

---

## 5. Processing Logic (Deterministic)

---

### Step 1: Entity Mapping
- Match parsed entities to canonical entities
- Ensure ID consistency

---

### Step 2: Relationship Extraction
- Traverse AST
- Identify:
  - method calls
  - imports/usages
- Create relationship entries

---

### Step 3: Vocabulary Normalization
- Tokenize names
- Apply normalization rules:
  - lowercase
  - snake_case conversion
- Map synonyms

---

### Step 4: Pattern Detection
- Apply rule-based detection:
  - naming conventions
  - structural signatures
- Assign pattern labels

---

### Step 5: Semantic Tagging
- Use deterministic rules:
  - folder structure
  - naming patterns
- Assign tags

---

### Step 6: Cross-Linking
- Link:
  - config keys → code usage
  - APIs → implementations

---

### Step 7: Validation
- Ensure:
  - no dangling relationships
  - consistent tagging

---

### Step 8: Output Preparation
- Merge into entity model
- Attach confidence + provenance

---

---

### 9.3 Edge Cases

* Ambiguous naming → lower confidence
* Missing config → skip linking
* Circular dependencies → flag

---

## 10. Failure Handling

### Failure Conditions

* Relationship resolution failure
* Pattern misclassification

### Strategy

* Partial enrichment allowed
* Mark low confidence
* Log inconsistencies

---

## 11. Evaluation Criteria

### Functional

* Relationship accuracy ≥ 90%
* Pattern detection accuracy ≥ 85%

---

### Performance

* Must scale to large graphs

---

### Test Scenarios

* Layered architecture repo
* Microservices repo
* Poorly structured repo

---

## 12. Constraints & Guardrails

* Prefer AST over heuristics
* No hallucinated relationships
* Deterministic tagging rules
* All enrichments must be explainable

---

## 13. Observability

Track:

* relationships created
* patterns detected
* tagging distribution

---

## 14. Security

* Do not expose sensitive config values
* Mask secrets in config linking

---

## 15. Versioning

* Pattern rules must be versioned
* Vocabulary must be versioned

---

## 16. Future Enhancements

* Machine learning-based pattern detection
* Cross-repository linking
* Code Property Graph (CFG/DFG)
* Feedback-driven vocabulary evolution
